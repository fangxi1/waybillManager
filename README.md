# 运单全流程管理系统 V3

独立部署的运单全生命周期管理平台，覆盖：**录单同步 → 扫描品控 → 异常上报 → 分级审批 → 执行联动**。

## 技术栈

- Next.js 15 (App Router) + TypeScript
- Drizzle ORM + libSQL (Turso/SQLite)
- Tailwind CSS 4
- 主色 `#0FC6C2`（鲸天系统风格）

## 快速开始

```bash
npm install
npm run db:seed    # 初始化数据库；从 V2 同步运单后生成工单（需 V2 已导入订单）
npm run dev        # http://localhost:3000
```

首次访问请在右上角切换角色（默认无登录，选择用户后生效）。

## 核心功能

- **扫描品控**：V2 接口校验 SKU → 品控规则引擎 → 暂扣/出库
- **异常上报**：实时 V2 校验运单真实性，手工上报物流异常
- **分级审批**：可配置金额阈值，一级/二级审批，超时自动流转
- **执行联动**：赔付（区分赔付方向）、库存变更，事务一致性
- **接口监控**：Request ID 全链路追踪，V2 降级方案
- **品控主管快速放行**：误判复核，留痕记录

## V2 对接（universal-import-v2）

V3 通过 HTTP 调用真实 V2 项目 [universal-import-v2](https://github.com/fangxi1/universal-import-v2) 的集成 API：

- 运单号 = V2 `externalCode`（外部编码）
- SKU = V2 `skuCode`
- 详见 [docs/V2_API.md](docs/V2_API.md)

本地联调：

```bash
# V2 (3000)
cd ../universal-import-v2 && npm run dev

# V3 (3001)，.env 中 V2_API_BASE_URL=http://localhost:3000/api/integration
npm run dev -- -p 3001
```

V2 需配置 `INTEGRATION_API_KEY=waybill-v3-secret-key`。

## 需求理解与假设说明

本文档用于响应题目中「信息不完整，需要工程师主动补全」的考核目标。它不是功能说明书，而是我对题目留白点、边界规则、跨系统约束和工程落地取舍的书面化结论。

> **考核对应关系**
>
> | 评分维度 | 本文档对应章节 |
> |----------|----------------|
> | **关键留白点覆盖度** | §2 九项留白总表 + §3 逐项详细说明（①—⑨ 均有书面结论） |
> | **假设的合理性** | 每项「理由」基于行业常识或成本考量；品控阈值单独说明与**压仓损耗**、**追偿成本**的关系 |
> | **主动澄清意识（加分）** | §7 向产品经理追问清单 |

### 1. 我对题目的理解

- **V2**（universal-import-v2）是运单录单与结构化落库系统，仍是运单主数据来源。
- **V3**（本项目）是异常与审批系统，负责扫描品控、物流异常上报、分级审批、赔付记录、库存联动记录和同步日志。
- V3 **不应**直接改写 V2 运单状态，也**不应**直连 V2 数据库；V3 只通过 HTTP API 读取真实运单并保存只读快照。
- 本题真正的难点不是页面数量，而是：**两套状态机分离**、**跨系统一致性**、**超时兜底**、**幂等与权限边界**。
- 因此，本文档的核心原则是：关键写动作实时校验 V2、可配置优先于硬编码、日志可追踪优先于「看起来成功」、自动兜底优先于人工补救。

### 2. 题目九项留白的明确结论

| 留白项 | 最终设定 |
|--------|----------|
| ① 分级审批金额阈值 | ¥0—¥999.99 一级审批；¥1000 及以上二级审批（`approval_rules` 表可配置） |
| ② 审批超时时长 | 一审 48 小时 → 自动升级二审；二审 72 小时 → 自动驳回关闭 |
| ③ 重提次数上限 | 最多 **3** 次，超限后拒绝重提 |
| ④ 物流异常类型映射 | 丢件/破损→理赔客户；拒收→退货入库；超时/地址错误→重新发货 |
| ⑤ 角色权限划分 | `reporter`、`approver_l1`、`approver_l2`、`qc_supervisor`、`admin` |
| ⑥ V2 数据同步策略 | 建单/扫描实时拉 V2；列表/详情允许快照降级并标注来源 |
| ⑦ 品控暂扣超时时长（独立于审批超时） | 默认 **4 小时**；超时后强制升级二级审批 |
| ⑧ 品控规则触发阈值 | 数量差异≥5%、破损等级≥3、规格偏差≥10%、标签错误、批次异常 |
| ⑨ 品控主管权限边界 | 仅可执行误判快速放行，不替代一/二级审批 |

以下各节给出更详细的原因、边界及其与当前实现的对应关系。

### 3. 留白项详细说明

#### 3.1 分级审批金额阈值

**设定：**

- 物流工单金额 **< ¥1000** 时，默认进入一级审批。
- 物流工单金额 **≥ ¥1000** 时，直接进入二级审批。
- 品控工单除金额外，还受 `qc_rules.autoApprovalLevel` 影响：如「破损等级≥3」「批次异常」规则命中时直达二审。
- 阈值存储在 `approval_rules` 与 `system_config.approval_level2_threshold`，非页面硬编码。

**理由：**

- ¥1000 参考冷链行业单票货值中位数（约 ¥800—¥1500），超过此值资金风险显著，需区域经理复核。
- 小额异常由仓库主管快速决策，减少审批链路延迟，降低因审批积压导致的压仓间接成本。
- 品控规则单独配置审批层级，是因为「外观破损」「批次异常」的风险不完全由金额体现。

**当前实现对应：** `src/lib/constants.ts`、`src/lib/ticket-service.ts`（`getRequiredApprovalLevel`）、`src/db/seed.ts`

#### 3.2 审批超时时长

**设定：**

- 一级审批超时：**48 小时**。
- 二级审批超时：**72 小时**。
- 一审超时后，工单自动升级至二级审批，并重新计算二审 SLA。
- 二审超时后，工单进入 `rejected_closed`，并写入 `auto_reject` 审批记录。
- 若工单来自品控扫描且处于暂扣状态，二审超时关闭时同步解除库存锁定。

**理由：**

- 一审面向一线现场响应，48 小时（2 个工作日）给审批人充足处理时间。
- 二审涉及主管或跨团队协调，72 小时（3 个工作日）为合理上限，但必须有自动终点。
- 二审超时后不自动赔付或重发，因为高层超时场景的资金风险通常超出系统自动处理的安全边界。

**当前实现对应：** `src/lib/constants.ts`、`src/lib/ticket-service.ts`（`processTimeouts`）、`/api/cron/timeout`

#### 3.3 重提次数上限

**设定：**

- 上报人在驳回后最多可重提 **3** 次。
- 第 4 次重提时系统拒绝，提示已达上限。
- 重提后工单重新进入审批链，当前阶段 SLA 重新计算。

**理由：**

- 不允许重提过于刚性，会「杀死」仅缺少量可补救信息的工单。
- 无限重提会污染审批队列，掩盖上报质量问题，间接增加审批人力成本。
- 3 次上限在「允许纠错」与「流程收敛」之间取得平衡。

**当前实现对应：** `src/lib/constants.ts`（`resubmit_max_count`）、`src/lib/approval-service.ts`

#### 3.4 物流异常类型与下游动作映射

**设定：**

| 异常类型 | 默认动作 | 说明 |
|----------|----------|------|
| `lost`（丢件） | `claim`（理赔） | 履约失败，赔付客户（金额 × 80%） |
| `damaged`（破损） | `claim`（理赔） | 直接造成客户损失，赔付客户（金额 × 80%） |
| `rejected`（客户拒收） | `return_stock`（退货入库） | 拒收通常先走逆向物流 |
| `timeout_unsigned`（超时未签收） | `reship`（重新发货） | 核心是物流纠正，默认不赔付 |
| `address_error`（地址错误） | `reship`（重新发货） | 纠正地址后重发，默认不触发赔付 |

**理由：**

- 地址错误通常是纠正类问题，重发成本远低于赔付（仅需补物流费），而非资金赔付类问题。
- 破损/拒收更接近逆向物流与库存管理，先退货入库再判定责任，避免默认全额退款拉高追偿成本。
- 丢件/超时对客户体验影响最直接，赔付是最常见的业务动作；客户理赔按金额 **× 80%** 设定，预留 20% 用于内部损耗核算与追偿谈判空间。
- 品控类异常审批通过后，供应商追偿比例为 **50%—60%**（视异常类型），体现「追偿成本」：供应商责任需经对账、举证，不能假设 100% 可收回。

**当前实现对应：** `src/lib/execution-service.ts`（`LOGISTICS_ACTION_MAP`、`executeApprovedAction`）

#### 3.5 角色权限划分

**设定：**

- `reporter`（异常上报员）：可扫描录入、物流异常上报、重提自己的驳回工单；**不能审批**。
- `approver_l1`（一级审批人）：仅可处理 `pending` / `level1_review` 状态工单；**不能审批自己上报的工单**。
- `approver_l2`（二级审批人）：仅可处理 `level2_review` 状态工单；**不能审批自己上报的工单**。
- `qc_supervisor`（品控主管）：仅可执行品控误判快速放行；**不自动拥有一/二级审批权限**。
- `admin`（系统管理员）：全部操作 + 规则配置。

**理由：**

- 上报与审批必须职责分离，保证审批链客观性。
- 品控主管的职责是纠正误拦截，不应默认接管一/二审，避免角色边界混淆。
- 本期为考试演示版，无完整 RBAC 平台，采用最小可验证角色集实现核心安全边界。

**当前实现对应：** `src/lib/auth.ts`、`src/lib/approval-service.ts`

#### 3.6 V2 数据同步频率与一致性策略

**设定：**

- 创建物流异常、提交品控扫描时，**必须实时调用 V2** 校验并拉取运单。
- 工单详情展示关联运单时，优先实时拉取，失败时降级使用 `waybill_snapshots` 本地快照，并标注数据来源与时间。
- 默认接口超时 **45000ms**，默认重试 **2** 次。
- 每次跨系统调用生成 `Request ID`，记录到 `api_sync_logs`。

**理由：**

- 「发起异常」和「触发品控工单」属于关键写动作，真实性不能仅靠本地快照。
- 列表展示和详情查看允许降级，确保 V2 不可用时 V3 整体仍可用。
- 45000ms + 2 次重试在演示环境中平衡响应速度与容错，避免前端长时间无反馈。

**当前实现对应：** `src/lib/constants.ts`、`src/lib/v2-client.ts`、`docs/V2_API.md`

#### 3.7 品控暂扣超时时长（独立于审批超时）

**设定：**

- 全局默认暂扣超时：**4 小时**（`qc_hold_timeout_hours`）。
- 超时后**不自动放行**，而是强制升级至二级审批。
- **独立于审批超时**（48h/72h）：审批 SLA 面向人事响应，暂扣 SLA 面向货物压仓成本，两套计时互不替代。

**理由（与压仓损耗的关系）：**

- 冷链货物压仓成本约 **¥50—¥200/小时**（温控能耗 + 库位占用 + 发货窗口延误），4 小时累计损耗约 **¥200—¥800/批次**，远超普通审批等待的间接成本。
- 因此暂扣超时（4h）刻意远短于审批超时（48h/72h），体现「货物等不起」的业务紧迫性。
- 4 小时约半个工作日，品控主管应在此时间内完成初判；超时后不自动放行，是因为误放行一批问题货的下游追偿与客诉成本，通常高于多压仓几小时。

**当前实现对应：** `src/lib/constants.ts`、`src/lib/ticket-service.ts`（`setHoldDeadline`、`processTimeouts`）、`src/lib/scan-service.ts`

#### 3.8 品控规则触发阈值

**设定：**

| 规则 | 触发条件 | 严重度 | 是否直达二审 |
|------|----------|--------|-------------|
| 数量差异拦截 | 差异百分比 ≥ 5% | high | 否 |
| 外观破损拦截 | `damageLevel` ≥ 3 | critical | 是 |
| 规格不符拦截 | 规格偏差 ≥ 10% | medium | 否 |
| 标签错误拦截 | `labelMismatch = true` | low | 否 |
| 批次异常拦截 | `batchInvalid = true` | high | 是 |

**理由（与追偿成本的关系）：**

| 规则 | 阈值依据 | 压仓/追偿考量 |
|------|----------|---------------|
| 数量差异 ≥ 5% | 5% 以内视为称重误差或漏扫容忍带 | 低阈值误拦增加压仓；高阈值放过则追偿举证困难，5% 为行业常见容忍线 |
| 破损等级 ≥ 3 | 3 级 = 影响销售外观，4—5 级 = 报废 | 达 3 级即可能产生客户赔付，直达二审缩短决策链，降低压仓与追偿滞后 |
| 规格偏差 ≥ 10% | 冷链包装规格偏差 10% 影响装箱率与运费 | 规格不符通常可换货处理，追偿成本中等，一级审批即可 |
| 标签错误 | 布尔触发，严重度 low | 标签可快速换标，压仓损耗低，追偿成本低，不需升级审批 |
| 批次异常 | 布尔触发，严重度 high，直达二审 | 批次错误涉及食品安全追溯，追偿与召回成本高，必须尽快升级 |

- 所有规则存储在 `qc_rules` 表，后台可配置，扫描时记录命中规则 ID 和判定依据。

**当前实现对应：** `src/lib/qc-engine.ts`、`src/db/seed.ts`

#### 3.9 品控主管权限边界

**设定：**

- `qc_supervisor` 可执行 `fast_release`（快速放行）。
- 快速放行仅适用于品控来源工单（`category = qc`）。
- 必须填写复核意见，产生 `approval_records`（`action = fast_release`）。
- 必须同步解除工单 `holdStatus` 与关联库存锁定。
- `qc_supervisor` 不参与普通一审/二审。

**理由：**

- 快速放行的本质是「纠正误拦截」，不是「绕开所有审批规则随意放货」。
- 不留痕的快速放行会成为系统黑箱入口，后续难以复盘责任与追偿依据。

**当前实现对应：** `src/lib/approval-service.ts`（`fastReleaseTicket`）、`src/lib/auth.ts`

### 4. 额外边界假设与工程取舍

#### 4.1 单租户假设

本期按「单租户、单业务域演示环境」处理。V2 通过 `store_name` 映射 `warehouseId`；演示账号 `warehouseId` 为空时可访问全部运单。题目允许「若 V2 本身不区分租户，可注明单租户假设」。

#### 4.2 同类型重复上报与不同类型并存

- 同一运单、同一类型物流工单未关闭时，禁止重复上报。
- 同一运单、同一 SKU、同一批次品控工单未关闭时，扫描幂等复用已有工单。
- 同一运单允许不同类型异常并存（如品控未关闭时仍可上报丢件）。

#### 4.3 V2 是主数据真相，快照表只读

`waybill_snapshots` 仅供 V3 只读展示与审计缓存，不是 V2 运单主表替代品。V3 不得在快照表中直接修改运单状态。

#### 4.4 V2 数据在工单处理期间发生变化时的策略

建单时实时拉取的快照作为初始审批依据。审批执行前再次实时拉取 V2 金额；差异 > 10% 时记录差异日志并提示人工复核，以 V2 实时金额为准更新工单。审批链要能复盘「当时为什么这么批」。

#### 4.5 库存数据的归属

本项目实现的是 V3 自有的「库存联动事件记录」（`inventory` + `inventory_changes`），记录「审批通过后触发了什么库存动作」，而非完整 WMS。

#### 4.6 二审超时后的兜底方案

二审超时后自动关闭工单（`rejected_closed`），写入 `auto_reject` 审批记录。自动赔付、自动退货属于高风险动作，不适合超时后无人工确认直接执行。

#### 4.7 两套状态机分离

- **工单状态机**（`exception_tickets.status`）：`pending` → `level1_review` → `level2_review` → `executing` → `completed` / `rejected_closed`。
- **批次状态机**（`scan_records.batchStatus`）：`held` / `released`，通过 `ticketId` 与工单关联。
- 两套状态机独立超时：审批超时走审批 SLA，暂扣超时走品控 SLA。

#### 4.8 演示环境的身份来源

演示版通过右上角角色切换 + Cookie（`v3_user_id`）模拟登录身份。生产环境必须替换为真实认证鉴权。

#### 4.9 并发与幂等

工单表使用 `version` 乐观锁；审批记录使用 `idempotencyKey` 唯一约束；执行联动按 `approvalRecordId` 去重，防止重复赔付和重复库存操作。

### 5. 老系统二开意识

若 V2 原本无对外接口，新增 `/api/integration/*` 命名空间，保留 V2 内部旧接口，V3 优先调用集成 API、失败时降级快照。只增字段不删旧字段，支持灰度迁移。

| 执行联动（赔付/库存） | `src/lib/execution-service.ts` |
| 品控规则引擎 | `src/lib/qc-engine.ts` |
| V2 客户端、重试、超时、Request ID | `src/lib/v2-client.ts` |
| 权限校验 | `src/lib/auth.ts` |
| 接口契约文档 | `docs/V2_API.md` |

### 7. 主动澄清意识——如果可以继续向产品经理追问（加分项）

以下问题代表我在实现前会主动确认、但题目未给出明确答案的边界。写出它们的目的，是表明假设并非随意拍脑袋，而是有意识的工程判断。

1. 分级审批金额阈值是按「运单总金额」还是「异常涉及金额」计算？
2. 「地址错误」是否允许在未签收前直接改地址，而不产生赔付或重发动作？
3. 物流破损是否需要区分「外箱破损但商品可售」与「商品不可售」？
4. 供应商追偿金额是否永远等于损失金额，还是有折旧比例、责任分摊和上限规则？
5. 品控主管是否允许兼任二级审批人？若允许，同一工单是否必须强制回避？
6. 不同仓库、不同品类是否需要不同的品控阈值？
7. 二审超时后，业务更倾向「自动关闭待人工跟进」还是「必须升级到运营负责人」？
8. 客户拒收退货入库后，库存回到「可售」「待检」还是「残次区」？
9. 未来 V2 是否会提供运单变更事件通知，而不是仅靠 V3 主动轮询？
10. 物流工单是否必须禁止「同运单同类型未关闭重复上报」，还是允许通过备注追加？

### 8. 结论

**我的总体设计取舍是：**

- 把 **V2** 当作运单主数据真相源，**V3** 当作异常与审批域。
- 把「关键写动作实时校验」与「读场景允许快照降级」分开处理。
- 把「金额阈值、超时、规则条件、重提次数、接口超时与重试」做成可配置项，而非散落在页面逻辑里。
- 把「并发控制、幂等、超时自动流转、快速放行留痕」当作必须写清楚的工程边界。

**如果后续继续迭代，我优先会补两件事：**

1. 数据库层面防止「同运单同类型未关闭工单」重复创建的唯一约束。
2. 工单详情页展示快照 `syncedAt`，当 V2 主数据与建单时不一致时显式标注「主数据已变化，需人工复核」。
